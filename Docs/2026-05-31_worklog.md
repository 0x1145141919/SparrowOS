## 决策录

### 1. NVMe 设备切换脚本

**背景**：01:00.0 (Sandisk WD SN5x0 2TB) 被 `vfio-pci.ids=15b7:5017` 固定由 vfio-pci 接管，用于 SparrowOS VM 直通。但开发文件系统时 host Linux 也需要访问这块盘。

**决策**：写 `nvme_dev_switch.sh`，通过 sysfs 热切换驱动绑定，不改动启动配置。重启后恢复 VFIO 默认。

**问题**：第一版 `die()` 只写 stderr，stdout 无输出导致用户以为脚本没反应。第二版修复且移除了 ANSI 转义码。第二版误判系统盘 `nvme0n1` 占用的 bug 也已修复，改为通过 `/sys/block/nvme*/device/address` 精确匹配 BDF。

**结果**：`user_utils/nvme_dev_switch.sh` — `status`/`host`/`vfio`/`toggle` 四种模式。

### 2. 文件系统方向

**背景**：自制文件系统供 SparrowOS 使用，同时在 Linux 上运行。

**决策**：
- B+ 树作为唯一索引结构（替代 ext 的 bitmap + 间接块指针 + htree）
- 文件数据块索引：per-file B+ tree，key=逻辑块号，value=物理块号
- 目录项：per-dir B+ tree，key=filename(xxhash)，value=inode
- 空闲块/空闲 inode 管理：全局 B+ tree
- 块大小固定 4096，匹配 NVMe 物理扇区
- Inline data：≤128B 的文件嵌入 inode，不经过 B+tree
- v1 不带 journal，依赖 fsck 恢复
- 分区表用 GPT，不做裸盘 FS

**Linux 驱动路径**：内核模块 ravenfs.ko（不走 FUSE），后期移植到 SparrowOS VFS。

### 3. 调试安全策略

**背景**：内核模块容易 panic，需要在不炸 host 的前提下迭代。

**决策**：
- 所有 B+ tree / 数据结构操作先在用户态（mkfs / btree_debug）验证
- 内核模块测试在 QEMU VM 中进行，禁止在 host 上直接 insmod
- 用 `-s` (GDB stub) + `add-symbol-file` 调试内核模块
- 真盘（01:00.0）通过 vfio-pci 直通进 VM，guest 内 mkfs 和 mount 直接写在真盘上

### 4. Host-VM 文件传输

**背景**：host 编译，VM 测试，需要传递 .ko 和工具。

**决策**：
- 走 SSH + SCP（避免改动硬件拓扑，不影响 SparrowOS 共用 VM）
- VMtest.sh 加 `hostfwd=tcp::2222-:22`，host:2222 → guest:22
- `sync_to_vm.sh` 一键编译 + 推送

### 5. VM 环境

**背景**：arch-root.qcow2 已有（32G，5.6G 占用），但不确定内部状态。

**决策**：暂不重建，先检查现有映像内容。如有必要用 pacstrap 重装最小 Arch + sshd。Guest 使用 host 内核 boot 以避免模块版本不匹配（方案待定）。

### 6. 规划态度

**共识**：探索阶段，不做过早规划。核心基础设施（切换脚本、VM 环境、同步工具）就位后，按 B+tree → mkfs → mount → readdir → read → write 渐进实现。
