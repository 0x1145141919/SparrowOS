# FreePagesAllocator::BuddyControlBlock (BCB) 文档 v2

## 1. 引言

`BuddyControlBlock`（简称 BCB）是 FreePagesAllocator 中的核心类，用于实现一个基于位图（bitmap）的二进制伙伴分配器（binary buddy allocator）。它管理一段连续的物理内存段（segment），以 4KB 页为最小单位，支持 power-of-2 大小的块分配和释放。BCB 设计用于内核级内存管理，支持最大 24 阶（order），但实际使用中通常限制在 20 阶左右以避免过大内存开销。

BCB 的主要目标是高效管理物理页的分配和释放，减少碎片化，通过位图编码空闲状态，并使用多条目缓存（per-order cache）和区间查找优化性能。整个系统假设物理地址（phyaddr_t）至少 4KB 对齐。

本文档基于当前源代码（FreePagesAllocator_BCB.cpp、mixed_bitmap.cpp、FreePagesAllocator.h）讲解 BCB 的核心数据结构、用途、编码方式以及核心算法。

## 2. 核心数据结构及其用途

BCB 使用紧凑的数据结构来跟踪内存状态，避免链表开销。核心结构围绕位图和索引构建，支持快速查找和更新。

### 2.1 mixed_bitmap_t

- **定义位置**：FreePagesAllocator.h 和 mixed_bitmap.cpp。
- **用途**：这是一个继承自 `bitmap_t` 的位图类，用于管理位（bit）级状态。BCB 中只有一个全局 `mixed_bitmap_t` 实例（`order_freepage_existency_bitmaps`），它统一管理所有 order 的空闲状态，而不是每个 order 一个位图。这提高了空间效率（总位图大小为 2^{max_support_order + 1} 位）。
- **关键成员**：
  - `uint64_t entry_count`：位图总条目数（位数）。
  - `uint64_t* bitmap`：底层 uint64_t 数组存储位图数据。对于未指定基地址的 BCB（通常为第一个 BCB），使用固定数组 `first_BCB_bitmap` 以避免动态分配。
  - `uint8_t* byte_bitmap_base`：字节级访问基址（用于调试/打印）。
  - `uint64_t bitmap_size_in_64bit_units`：位图占用的 64 位单元数量。
- **关键方法**：
  - `bit_set(idx, value)`：设置位为 true（1）或 false（0）。
  - `bit_get(idx)`：获取位值。
  - `find_free_in_interval(start_idx, interval_length)`：在指定区间内查找第一个为 1 的位（空闲位）。返回相对于 `start_idx` 的偏移，或 `0xFFFFFFFFFFFFFFFFULL` 表示未找到。实现使用逐 64 位块扫描 + `__builtin_ctzll` 优化。
  - `mixedbitmap_base_specify(bitmap_base_addr)`：指定或自动分配位图基地址。若参数为 0 且内部 bitmap 为空，则使用静态数组 `first_BCB_bitmap`。

### 2.2 order_freepage_existency_bitmaps

- **用途**：单一 `mixed_bitmap_t` 实例，编码所有 order 的空闲块状态。位图按 order 分段存储（通过 `order_bases` 分隔）。
- **编码方式**：见第 3 节。

### 2.3 uint64_t order_bases[DESINGED_MAX_SUPPORT_ORDER]

- **用途**：每个 order 在全局位图中的起始偏移（base）。允许将单一位图逻辑分为 per-order 段。
- **DESINGED_MAX_SUPPORT_ORDER**：设计最大 order 上限，固定为 24。这是数组维度的上限，实际实例的 `max_supprt_order` 可以更小。
- **初始化**：在 `corebcb_mixedbitmap_base_acclaim()` 中计算：
  - `order_bases[0] = 0`
  - `order_bases[order+1] = order_bases[order] + (1ULL << (max_supprt_order - order))`
  - 这确保 order k 有 2^{max_supprt_order - k} 个位，匹配二叉树结构（高 order 位少，低 order 位多）。
- **示例**（max_supprt_order = 20）：
  - order 0: 基址 0，长度 2^20 位（1M 位）。
  - order 1: 基址 2^20，长度 2^19 位。
  - ...
  - order 20: 基址 ≈ 2^21 - 1，长度 1 位。

### 2.4 多条目缓存机制

BCB 使用**多条目缓存**而非简单的单条 hint，以提高分配效率。

#### 2.4.1 suggest_order_free_page_index[DESINGED_MAX_SUPPORT_ORDER]

- **类型**：`uint64_t[PER_ORDER_CACHE_SUGGEST_COUNT]`，其中 `PER_ORDER_CACHE_SUGGEST_COUNT = 8`。
- **用途**：每个 order 的缓存池，记录最近释放的空闲块索引。
- **初始化**：构造函数中全设为 `INVALID_INBCB_INDEX`（~0ULL）。

#### 2.4.2 suggest_order_cache_cursor[DESINGED_MAX_SUPPORT_ORDER]

- **类型**：`uint8_t`
- **用途**：当前写入位置，用于循环覆盖策略。

#### 2.4.3 cache_insert(order, idx)

- **行为**：将索引插入对应 order 的缓存池。若缓存未满则追加；若已满则覆盖 cursor 指向的条目，cursor 循环前进。

#### 2.4.4 cache_pick(order, out_idx)

- **行为**：遍历缓存池中所有有效条目，找到第一个仍然空闲（位图中为 1）的索引。若找到则移除该条目并返回 true；若条目已失效（位图中为 0）则清除该条目并继续扫描。

### 2.5 struct BCB_statistics

- **用途**：收集运行时统计，用于调试和性能分析。
- **成员**：
  - `free_count[DESINGED_MAX_SUPPORT_ORDER]`：每个 order 的空闲块数。
  - `suggest_hit[DESINGED_MAX_SUPPORT_ORDER]`：缓存命中次数。
  - `suggest_miss[DESINGED_MAX_SUPPORT_ORDER]`：缓存未命中次数。
  - `alloc_times_success`：成功分配次数。
  - `free_times_success`：成功释放次数。
  - `alloc_times_fail`：分配失败次数。
  - `scan_count`：线性扫描次数（缓存未命中后触发）。
  - `fold_count_success`：合并成功次数。
  - `fold_count_fail`：合并失败次数。
  - `split_count`：分裂次数。
- **用途**：在分配/释放/分裂/合并时更新，帮助诊断碎片化或效率问题。可通过 `print_all_statistics()` 输出完整报告。

### 2.6 其他辅助成员

- `phyaddr_t base`：内存段起始物理地址（4KB 对齐）。
- `uint8_t max_supprt_order`：最大 order（构造函数参数，≤ DESINGED_MAX_SUPPORT_ORDER）。
- `spintrylock_cpp_t lock`：自旋锁，用于并发保护（BCB 级别的细粒度锁）。
- `bool is_splited_bitmap_valid`：预留字段，当前未使用。

### 2.7 调试/验证框架（REPLAY_MODE）

当定义 `REPALY_MODE` 宏时，启用额外的验证逻辑：

- **order_Internal_bitmap[DESINGED_MAX_SUPPORT_ORDER]**：辅助位图数组，用于跟踪分裂节点的内部状态。
- **replay_internal_init()**：初始化内部位图。
- **replay_internal_mark_split(order, idx)**：标记节点已分裂。
- **replay_internal_mark_free(order, idx)**：标记节点已释放。
- **replay_validate_tree_no_lock(tag)**：递归验证整个二叉树结构的一致性，包括：
  - 父空闲时子节点必须非空闲。
  - 父已分配时子节点状态一致性。
  - 内部位图与存在性位图不冲突。
- **replay_validate_tree(tag)**：带锁的版本。

该框架用于运行时检查位图二叉树结构是否被破坏，仅在调试/测试阶段启用。

## 3. 编码方式

BCB 使用位图编码内存状态，强调“存在性”（existency）：

- **位值含义**：
  - 1（true）：该索引对应的块是空闲的（free page exists）。
  - 0（false）：该块已分配、分裂或不存在。

- **索引计算**：
  - 对于 order k，位图偏移 = `order_bases[k] + idx`，其中 idx 是该 order 下的逻辑块索引（0 ~ 2^{max_supprt_order - k} - 1）。
  - 物理地址 = `base + (idx << (order + 12))`（因为 order 0 = 4KB = 2^12 字节）。

- **二叉树隐式编码**：
  - 位图模拟伙伴系统的二叉树：order k 的 idx 对应 order k-1 的 `2*idx` 和 `2*idx+1`。
  - 合并/分裂时，通过位操作（如 `idx ^ 1` 计算伙伴，`idx >> 1` 计算父）维护树一致性。

- **初始化编码**：
  - 初始时，整个段视为一个 order `max_supprt_order` 的空闲块：只设 `order_bases[max_supprt_order] + 0 = 1`，其他位为 0。

- **一致性保证**：
  - 释放时检查二叉树一致性（如父位不应已为 1），若违反则返回 fatal 错误。
  - 启用 REPLAY_MODE 时可进行完整树验证。

这种编码方式空间高效（位图总大小 ≈ 2^{max+1} 位 ≈ 2 * 总页数位），时间上依赖位操作和内置函数（如 `__builtin_ctzll`、`__builtin_clzll`）优化。

## 4. 核心算法

BCB 实现标准伙伴系统算法：分配时向上搜索 + 向下分裂；释放时标记 + 向上合并。

### 4.1 size_to_order(uint64_t size)

- **算法**：计算最小 order k，使得 2^k * 4KB >= size（向上取整）。
- **步骤**：
  1. `num_pages = (size + 4095) / 4096`（向上取整到页数）
  2. `next_pow2` = 使用位操作计算 num_pages 的下一个 2 的幂（处理 num_pages=0 的边界）
  3. `order = 63 - __builtin_clzll(next_pow2)`
- **用途**：标准化请求大小。

### 4.2 allocate_buddy_way(uint64_t size, KURD_t& result, uint8_t align_log2)

- **算法**：从目标 order 向上搜索空闲块，若找到高 order 块则分裂。支持对齐要求。
- **步骤**：
  1. `size_order = size_to_order(size)`
  2. `align_order = (align_log2 > 12) ? (align_log2 - 12) : 0`
  3. `order = max(size_order, align_order)`
  4. 参数校验：若 `size_order > max_supprt_order` 或 `align_order > max_supprt_order` 则返回错误
  5. 可选调用 `free_pages_flush()`（当前注释掉）进行一致性检查
  6. 对于 `i = order` 到 `max_supprt_order`：
     - 调用 `cache_pick(i, cached_idx)` 尝试缓存命中
     - 若命中：
       - 若 `i > order`，调用 `split_page(cached_idx, i, order)`
       - 标记目标 order 对应位为 0
       - 更新统计
       - 若对齐要求高于 size 要求，继续分裂到 `size_order`
       - 返回物理地址
  7. 缓存未命中：线性扫描各 order 的位图区间，调用 `find_free_in_interval`
  8. 若找到：执行类似逻辑
  9. 未找到：返回 `FAIL_REASON_CODE_NO_AVALIABLE_BUDDY`
- **优化**：多条目缓存减少扫描；区间查找支持局部性；对齐处理满足特殊硬件要求。

### 4.3 split_page(uint64_t splited_idx, uint8_t splited_order, uint8_t target_order)

- **算法**：递归向下分裂块，直到目标 order。
- **步骤**：
  1. 参数校验：`splited_order < target_order` 或 order 超出范围则返回错误
  2. 若 `splited_order == target_order`：返回成功（触底）
  3. 清除当前 order 的位（标记为已分配/分裂）
  4. 在 REPLAY_MODE 下标记分裂
  5. 递减当前 order 的 free_count
  6. 调用 `free_page_without_merge(1 + (splited_idx << 1), splited_order - 1)` 释放右子块
  7. 递增下一级 order 的 free_count
  8. 递增 split_count 统计
  9. 递归调用 `split_page(splited_idx << 1, splited_order - 1, target_order)`
- **注意**：分裂后，左子块用于继续分裂，右子块保持空闲并加入缓存。

### 4.4 conanico_free(uint64_t in_bcb_idx, uint8_t order)

- **算法**：释放并向上合并（coalesce）。
- **步骤**：
  1. 参数校验（order 范围、索引范围）
  2. 检查 double free：若位图中该位已为 1，返回 `FAIL_REASON_CODE_DOUBLE_FREE`
  3. 设置位为 1（空闲）
  4. 调用 `cache_insert(order, in_bcb_idx)`
  5. 递增 free_count
  6. `current_order = order`，`current_idx = in_bcb_idx`
  7. `while current_order < max_supprt_order`：
     - `buddy_idx = current_idx ^ 1`
     - 若伙伴位不为 1，break（无法继续合并）
     - 清除伙伴位和当前位（设为 0）
     - free_count[current_order] 减 2
     - `current_idx = current_idx >> 1`
     - 若父位已为 1（一致性检查失败），返回 fatal 错误
     - 设置父位为 1
     - 调用 `cache_insert(current_order + 1, current_idx)`
     - free_count[current_order + 1] 加 1
     - `current_order++`
  8. 递增 free_times_success
  9. REPLAY_MODE 下验证树结构
- **用途**：核心释放逻辑，确保最大合并。

### 4.5 free_buddy_way(phyaddr_t base, uint64_t size)

- **算法**：用户释放入口。
- **步骤**：
  1. 调用 `is_addr_belong_to_this_BCB_no_lock` 检查起始和结束地址是否都属于该 BCB
  2. 若不属，打印详细错误信息（包含地址范围对比）并返回 `FAIL_REASON_CODE_BASE_NOT_BELONG`
  3. `order = size_to_order(size)`
  4. `in_bcb_idx = (base - this->base) >> (order + 12)`
  5. 调用 `conanico_free(in_bcb_idx, order)`

### 4.6 free_page_without_merge(uint64_t in_bcb_idx, uint8_t order)

- **算法**：简单释放，不尝试合并。
- **步骤**：
  1. 设置位图中对应位为 1
  2. 递增 free_count[order]
  3. 调用 `cache_insert(order, in_bcb_idx)`

### 4.7 辅助算法

#### 4.7.1 cache_insert / cache_pick
见第 2.4 节。

#### 4.7.2 can_alloc(uint8_t order)
检查 order 及以上级别是否有空闲块（通过遍历 free_count）。

#### 4.7.3 free_pages_flush()
扫描位图，验证 `statistics.free_count[order]` 与实际位图空闲位数量是否一致。若不匹配，打印错误并设置 fatal 错误码（当前不自动修正，不触发 panic）。

#### 4.7.4 print 系列方法
- `print_basic_info_no_lock()`：打印 base、max_supprt_order、各 order free_count、缓存内容。
- `print_bitmap_info_no_lock()`：打印所有 order 的位图压缩信息。
- `print_bitmap_order_interval_compress_no_lock(order, base, length)`：区间压缩打印，将连续相同状态的位合并输出（如 `[0,127]=EXIST`）。
- `print_all_statistics()`：打印完整统计报告，包含缓存命中率、分配成功率、折叠成功率等。

### 4.8 未使用/预留功能

- **top_fold() ：实现了递归折叠逻辑，但当前未被调用。
正常分配释放逻辑不会产生拥有空闲邻居却不合并的稳定态，若此函数折叠成功则说明有非法状态，最好panic
- **is_splited_bitmap_valid**：成员变量存在但未使用，应该是要被迭代削减的设计

## 5. 并发模型

BCB 使用细粒度锁（`spintrylock_cpp_t lock`）保护每个 BCB 实例。FreePagesAllocator 层面维护多个 BCB 实例（`BCBS` 数组），分配时根据 NUMA/策略选择合适的 BCB，减少锁竞争。

## 6. 测试和验证

建议的测试方法：
- 创建 BCB（指定 base 和 max_support_order）
- 调用 `corebcb_mixedbitmap_base_acclaim()` 初始化位图
- 执行随机分配/释放操作，验证：
  - 无 double free
  - 无内存泄漏（所有分配最终可释放）
  - 地址范围始终在 BCB 管理范围内
- 启用 REPLAY_MODE 进行一致性验证
- 使用 `print_all_statistics()` 分析缓存命中率、碎片程度

## 7. 与 v1 文档的主要差异

| 方面 | v1 文档 | v2 实现 |
|------|---------|---------|
| hint 机制 | 单条 hint | 8 条目循环缓存 |
| 位图分配 | 仅第一 BCB 用静态数组 | 基地址为 0 时自动选择静态数组 |
| 调试框架 | 无 | REPLAY_MODE 完整验证 |
| top_fold | 描述为主动合并 | 已实现但未使用 |
| free_pages_flush | 校准计数 | 只检测不修正 |
| is_splited_bitmap_valid | 有效标志 | 未使用 |
| DESINGED_MAX_SUPPORT_ORDER | 未区分 | 设计上限 24 vs 实例 order |
| 地址归属检查 | 无 | 完整边界检查 + 错误日志 |
| 对齐分配 | 无 | 支持 align_log2 参数 |
| 统计报告 | 基础 | 完整的缓存命中率、成功率等 |
