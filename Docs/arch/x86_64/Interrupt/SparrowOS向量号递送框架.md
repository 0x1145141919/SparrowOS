# SparrowOS 向量号递送框架 (已废弃)

众所周知，x86的向量号以256为上限，我们通过向量号来进行中断/异常/ipi。idt把这个风马牛不相及的一堆机制押解在这牢笼中，而fred，虽然一定程度上把软中断和硬中断分开，但是还是需要特殊设计。由是，我围绕IDT以及FRED的向量号递送原理，以及实际上的工程实践，对软中断，ipi，异常，硬件中断，这四个原本应该拆伙过日子的，分门别类进行设计。
软中断：是x86上通过int x（x不为int3/int1那几个）实现，是用于实现各种执行流完整交出，获取完美上下文的调用，比如内核线程调用，用户线程系统调用，内核asm panic机制。由是这个是全局统一的机制，全局统一向量号，统一行为，对应到数据结构就是全局唯一数组soft_interrupt_functions。
ipi：此处的ipi特指系统语义级别的状态空间相当有限的ipi，比如tlb刷新，AP动工信号，重调度，下线信号。这种语义足够小，并且所有处理器通用，并且程序指针是否跑飞要具体情况具体分析，遂需要ipi_descrioptors数组，全局对那些事务统一向量号。
硬件中断：首先是正经的硬件中断，本身就用处理器信息，遂必须是per-cpu的，在每个CPU的gs_complex_t中。硬件中断，以及想让远程执行远端函数的ipi，都是通过这个机制。
当然，这四个必须用相关机制保证在递送后不能被混淆，此时就分idt/fred的机制来进行分拣
idt机制的idt第一入口，在0~31是异常直接进入，32～255是走向量号递送，通过all_vec_delivery进入。很明显，idt跳入的时候只带了向量号信息，必须通过向量号信息确认是什么，是软中断/IPI/硬件中断，由是我采取了死顺序优先非空跳转的机制——拿一个向量号，首先看对应的soft_interrupt_functions是否非空，非空就执行，不会返回（ud2埋雷）。其次是系统级ipi，执行后若ipi_descrioptors[vec].is_no_return为真则爆雷ud2。最后是硬件中断，执行res= self->tokens[vec].func(&self->tokens[vec])；必然会返回，不能自己跑飞。
由是idt下，软中断，系统级ipi，不能自助分配。并且软中断向量集，系统级ipi向量集，硬中断向量集，两两之间不可以有交集造成重复。
而在FRED中，SS.type可以直接分离异常/软中断/硬件中断，但是硬件终端中会杂糅ipi与硬件终端，这是牢英刻意偷懒，在硬件中断中，我们还是要先尝试硬件中断，再硬件中断。由是fred的软中断表可以自由分配，但是系统级ipi表必须在Init中进行初始化，并且不得自由分配
